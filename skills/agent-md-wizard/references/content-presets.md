# Content Presets

在用户没有提供明确措辞时，优先复用下面这些中文预设，再结合仓库探测结果做轻量调整。

## 项目定位预设

- `application`：这是一个面向终端用户或内部使用者的应用项目，重点关注交互体验、页面流程、发布稳定性和回归风险。
- `service`：这是一个提供接口、任务或后端能力的服务项目，重点关注契约兼容性、错误处理、可观测性和资源安全。
- `library`：这是一个供其他系统或团队复用的库项目，重点关注 API 稳定性、版本兼容、示例完整度和可发布性。
- `full-stack`：这是一个同时包含前端与后端能力的全栈项目，重点关注跨层改动的一致性、接口联动与端到端验证。

## 编码风格预设

- 优先沿用仓库已有目录结构、命名方式和依赖约定。
- 优先最小改动闭环，再补充必要说明，不把“顺手重构”混进本次任务。
- 遇到复杂逻辑先补边界和失败路径，再考虑抽象整理。
- 新增配置、脚本或依赖前先确认收益，避免让仓库复杂度持续累积。

## Review 关注点预设

- 需求是否真实落地，是否只修了表象问题。
- 是否引入回归、兼容性问题或隐式行为变化。
- 错误处理、超时、重试、并发与资源释放是否完整。
- 配置、迁移、权限和敏感信息是否处理安全。

## 测试门槛预设

- 低门槛：提交前至少运行与改动直接相关的检查；若无法运行需明确原因。
- 标准门槛：提交前运行核心 lint 与测试，并说明影响范围和未覆盖项。
- 高门槛：除 lint/test 外，还要补充回归场景、构建验证或端到端说明。

## 风险控制预设

- 敏感信息：禁止泄露密钥、Token、证书、客户数据和内部地址。
- 迁移变更：涉及数据库、接口契约、配置格式或基础设施时，先说明影响、验证和回滚方案。
- 生成文件：仅在可复现且明确需要时提交，否则说明生成来源。
- 协作边界：默认不覆盖他人未确认的修改，发现冲突先对齐再落盘。
- 危险命令：`git reset --hard`、`git checkout --`、`rm -rf`、破坏性数据库操作、不可逆基础设施销毁命令。
